# AGENTS.md

## Purpose

This repository contains a Tauri v2 workspace for a desktop emoji picker and a custom XDG portal plugin.

## Localisation and Spelling

**REQUIREMENT:** All UI strings, code variables, comments, commit messages, pull request descriptions, and documentation MUST use **Canadian English** spelling.

Apply this to code and naming as well, unless exact spelling is required by tooling, third-party APIs, platform interfaces, or external protocols.

Examples:

- `colour` not `color`
- `centre` not `center`
- `neighbour` not `neighbor`
- `cancelled` not `canceled`
- `licence` (noun) / `license` (verb)

## Repository Layout

- `apps/emoji-picker/`: React + TypeScript frontend for the desktop app.
- `apps/emoji-picker/src-tauri/`: Rust/Tauri backend for the desktop app.
- `plugins/xdg-portal/`: Workspace plugin crate (`tauri-plugin-xdg-portal`) and guest JS bindings.
- `plugins/xdg-portal/guest-js/`: TypeScript guest API package consumed by the app.
- `plugins/xdg-portal/permissions/`: Plugin permissions and autogenerated permission artifacts.

## Workspace and Scripts

- Package manager: `pnpm` workspace.
- Root scripts:
  - `pnpm dev` runs `@emoji-picker/emoji-picker`.
  - `pnpm build` builds `@emoji-picker/emoji-picker`.
- Rust workspace members:
  - `apps/emoji-picker/src-tauri`
  - `plugins/xdg-portal`

## Tauri v2 Rules

- Use Tauri v2 capabilities (not v1 allowlist patterns).
- Keep app capabilities in `apps/emoji-picker/src-tauri/capabilities/`.
- Ensure plugin command permissions used by the app match real permission identifiers.
- If plugin commands change, regenerate/refresh permission metadata via plugin `build.rs`-driven generation.

## XDG Portal Plugin Rules

- Plugin commands are registered in `plugins/xdg-portal/build.rs`.
- Prefer autogenerated command permissions under `plugins/xdg-portal/permissions/autogenerated/commands/`.
- Do not add redundant manual command permission files when autogenerated permissions already cover them.
- Keep `plugins/xdg-portal/permissions/default.toml` minimal and safe by default.

## Coding Conventions

- Prefer direct imports; avoid adding barrel re-export files unless explicitly requested.
- Keep TypeScript and Rust APIs strongly typed.
- Keep platform logic explicit (`Wayland` vs `X11` detection/handling in backend).

## Commit Messages

**REQUIREMENT:** Use Conventional Commits format (for example: `feat: ...`, `fix: ...`, `docs: ...`, `test: ...`).

- Use `test:` for test-related changes, including fixes to tests themselves (use `fix:` only for application code fixes).

Body requirements:

- Explain what changed and why.
- Use markdown formatting where useful (`**bold**`, `_italics_`, `code`, bullets).
- Do not use markdown headings in commit bodies.
- Keep each commit message focused on that specific commit.

Shell safety:

- Do not pass markdown-heavy commit bodies directly via `git commit -m \"...\"` when they contain backticks, `$()`, or shell-sensitive characters.
- Prefer writing the message to a file and commit with `git commit -F <file>`.
- Verify the stored message with `git log -1 --pretty=fuller`.

## Pull Request Titles

**REQUIREMENT:** PR titles must be human-readable summaries of the behavioural change.

- Start with a capital letter.
- Do not use Conventional Commit prefixes in PR titles (no `feat:`, `fix:`, `chore:`).
- Describe outcome/behaviour change, not implementation process.
- Keep title style consistent across an open PR stack.

## Licence and Copyright

**REQUIREMENT:** All source code files (`.rs`, `.ts`, `.tsx`, `.js`, etc.) must include a licence/copyright header as the first content in the file.

Header format:

```text
// Brief one-line summary of what this file does
//
// (c) Copyright 2026 Liminal HQ, Scott Morris
// SPDX-License-Identifier: Apache-2.0 OR MIT
```

Rules:

- Place the header before `use`, `mod`, `import`, or other code.
- Do not add headers to generated files, config files, or documentation files.
- If you touch an existing source file without a header, add one as part of the change.

## Git Workflow

- Do not push or force-push unless explicitly requested.
- Use focused commits with clear messages describing what changed and why.
