# Building Emoji Nook With AI: What Worked, What Didn't, and What Surprised Us

**TL;DR** — Emoji Nook was built collaboratively with Claude, Anthropic's AI assistant, across a handful of conversations in Claude Code. The human steered design, validated behaviour, and provided taste. The AI handled implementation, debugging, documentation, and git workflow. One AI wrote the code, another reviewed it. Here's what that process actually looked like.

---

## The Workflow

Emoji Nook wasn't built by pasting prompts into a chat window and copying code back out. It was built conversationally, using [Claude Code](https://docs.anthropic.com/en/docs/claude-code), Anthropic's CLI tool that gives Claude direct access to the project's files, terminal, and git history.

The pattern looked like this: Scott described what he wanted — a compact emoji picker that blends into the Linux desktop, uses `xdg-desktop-portal` for theme detection and Wayland shortcuts, injects emoji via a clipboard shuffle. Claude read the spec, scaffolded the workspace, and started building. When something broke, Scott provided a screenshot or described the symptom, and Claude diagnosed and fixed it. When a design decision was needed — how the shortcut should toggle, whether to close on select, how to structure the portal plugin — Scott made the call, and Claude implemented it.

Context accumulated across each conversation. By the midpoint, Claude understood the permission system, the display server routing, the Frimousse component model, and the project's conventions well enough to make reasonable implementation choices without being told every detail. That's the key difference from prompt-and-paste: the AI learns the codebase as you work.

The whole arc — from an empty directory to a working picker with native theming, global shortcuts, emoji injection, settings persistence, CI pipelines, release automation, and architecture documentation — happened across a handful of Claude Code sessions.

## What AI Was Good At

### Rapid Scaffolding

Standing up a Tauri v2 workspace with a React 19 frontend, a Rust backend, and a custom plugin in a separate workspace crate is not trivial. There's `tauri.conf.json`, Cargo workspace members, pnpm workspace configuration, plugin guest bindings, permission manifests, capability files, and build scripts that all need to agree with each other. Claude set this up quickly, and — critically — knew how to fix the wiring when pieces didn't connect.

### Cross-Referencing a Working Project

When the custom portal plugin's permission system broke (Tauri v2's autogenerated capabilities weren't populating), we were stuck. The error messages were unhelpful. The docs described the system but not the exact TOML format required. Claude read a working reference app (`~/source/threshold`) that had a functioning Tauri v2 plugin, compared the permission file formats, spotted the difference (`[default]` section headers vs `[[permission]]` array-of-tables), and applied the fix. This is exactly the kind of debugging that benefits from being able to read and compare multiple codebases simultaneously.

### Reading Unfamiliar Source Code

Frimousse is a headless React emoji picker — it gives you unstyled components and lets you build the UI yourself. That's great for flexibility, but it means the library's internal behaviour isn't always documented. When we needed to understand how focus and keyboard navigation worked inside the emoji grid, Claude read Frimousse's minified source code and identified the internal model: `onFocusCapture` on the list element, a `data-focused` attribute on the active emoji, and `document.addEventListener` for arrow key handling. That understanding informed how we built the category navigation bar and why we needed to suppress certain focus outlines.

### Debugging From Screenshots

Scott provided screenshots showing visual bugs — a thick blue focus rectangle behind the emoji grid, theme colours not applying correctly, layout issues with the category bar. Claude diagnosed the CSS causes from the screenshots without needing to reproduce the issue locally. The focus outline, for instance, was traced to `tabIndex={0}` on the viewport element causing WebKitGTK to render its default outline, which was unnecessary since Frimousse already provides visual focus feedback on the active emoji.

### Documentation

Claude produced architecture documentation with Mermaid diagrams, animated SVG diagrams for the data flows (emoji selection pipeline, clipboard shuffle, theme detection, display server routing), implementation plans with phased milestones and review gates, and a Linux setup guide. The animated SVGs were inspired by the visual style of other open-source projects' documentation — Claude adapted the approach to fit the project's specific architecture.

### Git Workflow

Conventional commit messages, interactive rebasing to sort autogenerated permission files into the correct commits, batch-rewriting commit messages to match the project's style conventions, and creating pull requests with the repository's format rules (summary sections, test plan checklists, proper labels). Git housekeeping is tedious work that benefits enormously from automation, and Claude handled it fluently.

## What Needed Human Judgement

### Design Philosophy

The core decision — "blend in, don't stand out" — shaped everything. It determined that the picker should read the desktop's actual theme tokens from `xdg-desktop-portal`, not ship its own colour scheme. It determined the compact 370x380 overlay form factor. It determined that the app should match Cantarell with 12px radii on GNOME and Noto Sans with 6px radii on KDE. An AI can implement a design philosophy, but choosing one requires understanding what makes a tool feel native on someone's desktop. That's taste, not engineering.

### Timing and Feel

The clipboard shuffle has hardcoded sleep durations: 100ms for focus to settle on the target app after the picker hides, 200ms for the paste to complete before restoring the clipboard. These numbers need real-world testing on different compositors, different hardware, and different application toolkits. Too short and the paste fails silently. Too long and the user notices the delay. Claude could implement the mechanism, but tuning it requires running it on actual desktops and feeling whether it's right.

### UX Decisions

When should the picker close — on every selection, or stay open for rapid-fire entry? Should the shortcut toggle the picker or just show it? Should blur-to-hide be suppressed while the settings panel is open (yes — native dropdown menus trigger blur events)? These are interaction design questions that require thinking about how the tool fits into someone's workflow. They're small decisions, but they compound into the difference between a tool that feels considered and one that feels like a demo.

### Architecture Boundaries

Structuring the portal plugin as a separate workspace crate (rather than inline code in the app) was a deliberate choice to make it reusable by other Tauri apps that need portal access. The decision to leak the Wayland session handle with `std::mem::forget` — rather than building a more complex lifetime management system — was a pragmatic tradeoff that prioritised shipping over elegance. These are architectural calls that depend on project goals, not just technical correctness.

### Prioritisation

Which bugs to fix now vs defer. What goes in the first PR vs the next branch. Whether to implement the RemoteDesktop portal injection now or ship with the clipboard shuffle and iterate. These decisions require understanding the project's trajectory and the cost of delay, which is context an AI doesn't have.

## Debugging Stories

### The Tauri v2 Permission Format

Tauri v2's plugin permission system uses auto-generated TOML files that must follow a specific format. Our plugin's `default.toml` used `[[permission]]` (an array of tables), but the framework expected `[default]` (a single table with an `identifier` field and a `permissions` array). The error messages didn't point to this difference. The documentation described the *concept* of capabilities but not the exact file format.

The fix came from cross-referencing. Claude read the working plugin in `~/source/threshold`, compared the TOML structures line by line, identified the format discrepancy, and applied the correct structure. Without a working reference to compare against, this could have been hours of trial and error.

### The Viewport Focus Outline

Adding `tabIndex={0}` to the Frimousse viewport was necessary for keyboard navigation — without it, the emoji grid couldn't receive focus. But WebKitGTK rendered a thick blue focus rectangle around the entire viewport element, visible behind the emoji grid. Scott sent a screenshot showing the issue.

Claude diagnosed it: WebKitGTK's default focus style was being applied to the viewport's outline, and since the viewport was the full size of the emoji grid, the outline appeared as a large blue rectangle. The fix was straightforward — suppress the outline on the viewport via CSS. Frimousse already highlights the focused emoji with its own visual indicator, so the browser's default outline was redundant.

### Category Scroll-To

The category navigation bar needs to scroll the emoji grid to the right section when you click a category tab. The initial implementation used `offsetTop` and `offsetParent` to calculate the scroll position of category headers. It worked in some cases but produced wrong offsets in others.

The problem was Frimousse's DOM nesting. The category headers weren't direct children of the scroll container — they were nested inside intermediate elements, which meant `offsetTop` was relative to the wrong parent. `getBoundingClientRect()` solved it cleanly because it returns absolute viewport coordinates regardless of DOM nesting depth.

### The IntersectionObserver Timing

The category bar highlights the currently visible category as the user scrolls through the emoji grid. The implementation used an `IntersectionObserver` to watch category header elements and update the active tab as headers entered or left the viewport.

The bug: the observer was set up once on mount, but Frimousse renders its emoji data asynchronously. When the observer initialised, some category headers didn't exist in the DOM yet. The observer never saw them appear, so scrolling past those categories didn't update the highlight.

This one was caught in code review (see below), not during development. The fix was a `MutationObserver` that watches for new header elements and adds them to the `IntersectionObserver` as they appear.

## The AI-Assisted Code Review Loop

Here's the part that surprised us.

After the main implementation PR was ready, [Codex](https://openai.com/index/codex/) (OpenAI's code agent, acting as a code reviewer) reviewed the pull request and left two comments. Both were valid:

1. **The `IntersectionObserver` only ran once on mount**, missing category headers that Frimousse rendered asynchronously after the initial paint. The fix was adding a `MutationObserver` to detect new headers and register them with the intersection observer dynamically.

2. **The theme hook forced light mode when the portal returned `no-preference`**. The original code treated `no-preference` as "use light", but the correct behaviour is to fall back to the browser's `matchMedia('(prefers-color-scheme: dark)')` query, which respects the user's system-level preference even when the portal doesn't express one.

Claude fixed both issues and replied to the PR comments explaining the changes.

The pattern here is worth noting: one AI wrote the code, a different AI reviewed it, and the first AI fixed the issues the second one found. The human's role was to read both the review comments and the fixes, confirm they made sense, and approve the changes. Neither AI needed the human to explain the bugs — one found them, the other understood and fixed them.

This isn't a replacement for human code review. Both AI reviewers missed things a human would catch (and vice versa). But as a complement to human review, having a second AI perspective caught real bugs that shipped fixes for. Different models, trained differently, notice different things.

## Takeaways

**AI dramatically accelerates implementation, but it doesn't replace design thinking.** The parts of Emoji Nook that make it feel like a native Linux tool — the theme token system, the compact overlay form factor, the philosophy of blending in — came from human design decisions. The parts that made it *ship* — the Tauri workspace wiring, the Rust/React IPC layer, the permission system debugging, the documentation — were dramatically accelerated by AI.

**The conversational model works better than prompt-and-paste.** Context accumulates. By the middle of a session, Claude understood the codebase well enough to make implementation choices that were consistent with the project's patterns. That continuity matters more than raw code generation speed.

**Having a reference project is invaluable.** The Tauri v2 permission format issue was only solvable by comparing against a working plugin. Frameworks move fast, documentation lags, and error messages are often unhelpful. Being able to point the AI at a working example and say "figure out what's different" is one of the most productive patterns we found.

**AI-written code benefits from AI code review.** Codex caught two real bugs that Claude's implementation missed. Different models have different blind spots. Using one AI to write and another to review is a lightweight way to catch issues that a single model's perspective would miss.

**The human's role shifts, but doesn't shrink.** You stop writing boilerplate and start making decisions. You stop debugging CSS specificity and start evaluating whether the interaction feels right. You stop writing commit messages and start deciding what goes in which commit. The work changes shape, but the judgement — the taste, the prioritisation, the "is this actually good?" — stays human.
